Skechers Email Audit

Your 20% Off Welcome Offer is waiting!

FromSKECHERS <no-reply@emails.skechers.com>
Received2026-03-13 21:20 UTC
Score5/10

Executive Summary

This is the third touch in Skechers' non-member welcome series, leading with a personalized 20% off promo code (WEL20-C39C3FPRRN) valid through April 10, 2026. The primary offer is clear and the code is unique per recipient — that's the floor, and it's met. Everything else undercuts it. The email carries a critical merge error that breaks the redemption URL in legal copy, a blanket HTTP image infestation that will cause broken images in strict clients, zero product content to anchor intent, and a bottom half so loaded with utility modules (SMS opt-in, app download, curbside, BNPL x2, store locator, Plus pitch) that the conversion hierarchy collapses into noise. For a welcome series reaching a non-member for the third time, this email is not working hard enough.

The personalized code and offer deadline create urgency, but the broken URL in T&C copy, extensive HTTP asset exposure, and absence of product imagery mean this email is leaving money on the table from both a technical and a merchandising standpoint.

What’s Working

  • Unique promo code per recipient — WEL20-C39C3FPRRN is individualized, reducing abuse risk and enabling downstream attribution.
  • Offer clarity — Subject line, preview, and (presumably) hero all align on the same message: 20% off, act now.
  • Hard expiry date — April 10, 2026 is a concrete deadline, which helps urgency even if it's three-plus weeks away.
  • Size-filtered navigation — The email includes pre-filtered size links for Women, Men, and Kids (e.g., /women/shoes/?prefn1=size&prefv1=7%7C7.5). When personalized to a known size, this reduces click-to-cart friction. Execution is unclear without knowing whether sizes are actually personalized here.
  • Multi-category nav — Women, Men, Kids, Clothing, New Arrivals, Sale — standard but functional coverage.
  • BNPL presence — AfterPay and Klarna are both represented, which supports conversion for price-sensitive new customers.

What’s Weak

  • No product content — This is a welcome email for a shoe brand and there are zero product images, no featured styles, no bestsellers. A 20% off offer with nothing to shop is an incentive floating in a vacuum.
  • Utility module overload — SMS opt-in, app download, curbside pickup, BNPL (twice), store locator, and a Skechers Plus pitch all appear in the lower half of a single email. For a third-touch non-member who hasn't converted yet, this is noise, not value.
  • Skechers Plus is buried — The Plus membership pitch ("FREE SHIPPING + RETURNS ON ALL ORDERS") lands after five other utility modules. For a non-member welcome series, Plus enrollment is arguably the second-highest conversion goal after first purchase — it deserves a better position.
  • HTTP image assets — The Skechers logo, hero image candidates, app store badges, social icons, AfterPay logo, and tracking pixel all load over HTTP. Modern email clients (Gmail, Outlook, Apple Mail with image blocking) will either block or proxy these, producing broken image rendering for a significant share of recipients.
  • Alt text gaps — Multiple images have no alt text, meaning broken-image fallbacks show nothing. For recipients with images off, the email degrades to near-blank in key sections.
  • No visible personalization beyond the code — Third touch in a welcome series and the only personalized element appears to be the promo code. First name, browsing behavior, category preference — none visible.

Recommendations

  • Add product content to the hero section. Show 3–6 bestselling or trending styles with imagery, names, and prices. The discount is the hook; product is the reason to click.
  • Fix the HTTP image assets immediately. Update all http://image.emails.skechers.com src URLs to HTTPS across the template. This is a systemic template issue, not an isolated one — the same problem affects logo, app badges, social icons, and utility images.
  • Add alt text to all images. Even a generic fallback ("Shop Women's Shoes") preserves the call to action when images are blocked.
  • Fix the merge error in legal copy. The T&C block contains http://www.skechers.com/www.Skechers.com — a doubled-domain template variable that failed to resolve. This is a broken URL in visible legal copy. Fix the merge tag or use a plain-text URL in the fallback.
  • Collapse the utility module stack. Pick two: BNPL and one of (SMS / App / Plus). Move Plus higher if the goal is member acquisition alongside first purchase. The current seven-module bottom section dilutes everything.
  • Use the offer expiry for urgency pacing. Three weeks is long enough that urgency isn't felt. Consider adding a day-count or "X days left" callout near the code.
  • Personalize beyond the code. By touch 3, behavioral or demographic data should be available to surface a relevant category (e.g., "We think you'll love these Women's styles"). A generic shop-all CTA for a third-touch non-member is a missed signal.

Bottom Line

A structurally sound welcome offer undermined by template debt, zero product content, and module bloat. The promo code does its job; the email around it doesn't. Fix the HTTP assets and merge error first (both are table-stakes technical issues), then rebuild the lower half with focus. This email is reaching non-members who've now seen three touches without converting — it needs to earn the click, not just announce the discount.

Bugs / friction / clarity issues

CRITICAL: Merge error in T&C copy — http://www.skechers.com/www.Skechers.com is a doubled domain from a failed template variable substitution. The intended URL (www.Skechers.com) appended to the base URL instead of replacing a variable.

HIGH: 10+ image assets served over HTTP, including the Skechers logo and key utility images. Will render broken in image-blocking or HTTPS-strict email clients.

MEDIUM: Multiple images lack alt text — broken image fallback shows nothing, degrading the experience for images-off recipients.

LOW: Tracking pixel (o.gif) served over HTTP with no alt text — standard but worth noting alongside the broader HTTP pattern.

INFO: Plain-text version is 72% URLs by character count — this is structurally expected but reinforces that the plain-text fallback provides near-zero readable content for text-only clients.
